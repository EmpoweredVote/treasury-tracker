    Create the directory `supabase/functions/givebutter-webhook/` and write `index.ts` with the following behavior:

    **Imports:**
    Use `npm:@supabase/supabase-js@2` (current Supabase Deno convention — npm: prefix, not esm.sh).

    **Environment variables (read at top, outside the handler):**
    - `GIVEBUTTER_SIGNING_SECRET` — from `Deno.env.get()`
    - `SUPABASE_URL` — from `Deno.env.get()` (automatically injected by Supabase)
    - `SUPABASE_SERVICE_ROLE_KEY` — from `Deno.env.get()` (automatically injected; use service role, NOT anon key)

    **Supabase client (module-level, created once):**
    Use `createClient(SUPABASE_URL, SUPABASE_SERVICE_ROLE_KEY, { db: { schema: 'treasury' } })`.
    Service role is required to bypass RLS and write to the treasury schema.

    **`timingSafeEqual` helper function:**
    Deno does NOT have `crypto.timingSafeEqual` (that is Node.js-only). Implement via manual XOR reduction:
    ```typescript
    function timingSafeEqual(a: string, b: string): boolean {
      const enc = new TextEncoder()
      const aBytes = enc.encode(a)
      const bBytes = enc.encode(b)
      if (aBytes.length !== bBytes.length) return false
      let diff = 0
      for (let i = 0; i < aBytes.length; i++) diff |= aBytes[i] ^ bBytes[i]
      return diff === 0
    }
    ```

    **`Deno.serve` handler — complete request flow:**

    1. Reject non-POST: return 405 Method Not Allowed.

    2. Read raw body FIRST (CRITICAL — must happen before JSON.parse):
       `const rawBody = await req.text()`
       Do NOT call `await req.json()` at any point — that consumes the body and may normalize whitespace, breaking signature verification.

    3. Signature verification:
       - Get header: `const sig = req.headers.get('Signature') ?? ''`
       - If `!sig` or `!GIVEBUTTER_SIGNING_SECRET`: return 401 Unauthorized.
       - Verify: `if (!timingSafeEqual(sig, GIVEBUTTER_SIGNING_SECRET))` return 401 Unauthorized.
       - NOTE on algorithm uncertainty: Per research findings, GiveButter's docs say "Signature header contains the same value as your dashboard secret" — implement as direct string comparison first. If the $1 test shows mismatches in logs, the next iteration adds HMAC-SHA256. The code structure must make this swap trivial (keep verification logic in one function).

    4. Parse JSON:
       ```typescript
       let payload: { type?: string; data?: Record<string, unknown> }
       try {
         payload = JSON.parse(rawBody)
       } catch {
         return new Response('Bad Request', { status: 400 })
       }
       ```

    5. Event type filter:
       If `payload.type !== 'transaction.succeeded'`: return 200 OK silently (no log needed).

    6. Extract transaction fields from `payload.data`:
       ```typescript
       const tx = payload.data ?? {}
       const externalId = String(tx['id'] ?? '')
       const amount = Number(tx['amount'] ?? 0)
       const firstName = String(tx['first_name'] ?? '')
       const lastName = String(tx['last_name'] ?? '')
       const transactedAt = String(tx['transacted_at'] ?? '')
       ```
       Log ALL of these at the start of processing a transaction.succeeded event:
       ```typescript
       console.log('givebutter-webhook received transaction.succeeded', {
         externalId, amount, transactedAt,
         rawAmount: tx['amount'],   // log raw to confirm cents vs dollars in $1 test
         signatureHeader: sig,       // log to confirm algorithm in $1 test
       })
       ```
       These logs are the primary diagnostic tool for the $1 test.

    7. Guard against missing external_id:
       If `!externalId`: log a warning and return 200 (silently discard — can't dedup without an ID).

    8. Resolve category IDs dynamically (NEVER hardcode UUIDs):
       Query `budgets` table for the most recent revenue budget for EV (look up municipality by name 'Empowered Vote', dataset_type='revenue', order by fiscal_year DESC, limit 1).
       Then query `budget_categories` for `name IN ('Donations', 'Give Butter')` within that budget_id.
       Extract `leafCatId` (Give Butter, depth=1) and `parentCatId` (Donations, depth=0).
       If either is missing: log an error and return 500 (this is a configuration error, not a bad request).

    9. Build description:
       ```typescript
       const description = firstName
         ? `Donation from ${firstName} ${lastName}`.trim()
         : 'GiveButter donation'
       ```

    10. Call the Postgres RPC:
        ```typescript
        const { error } = await supabase.rpc('record_givebutter_donation', {
          p_external_id: externalId,
          p_leaf_category_id: leafCatId,
          p_parent_category_id: parentCatId,
          p_budget_id: budgetId,
          p_description: description,
          p_amount: amount,   // units unconfirmed; log above will reveal during $1 test
          p_date: transactedAt.slice(0, 10) || new Date().toISOString().slice(0, 10),
          p_vendor: 'GiveButter',
        })
        ```

    11. Handle RPC response:
        - If `error`: log the error details, return 500 Internal Server Error (GiveButter will retry).
        - If no error: log success with externalId, return 200 OK.

    Wrap the entire handler in a try/catch — any uncaught exception should log and return 500.

    Anti-patterns to avoid (from research):
    - Do NOT call `await req.json()` — use `req.text()` then `JSON.parse()`
    - Do NOT hardcode category UUIDs
    - Do NOT use `Deno.env.get('SUPABASE_ANON_KEY')` — use SERVICE_ROLE_KEY
    - Do NOT use `crypto.timingSafeEqual` — it does not exist in Deno; use the XOR pattern above
