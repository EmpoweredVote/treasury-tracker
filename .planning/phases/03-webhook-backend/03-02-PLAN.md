<objective>
Create the `treasury.record_givebutter_donation` Postgres function that atomically deduplicates and writes a GiveButter donation as a budget line item while updating all three pre-aggregated total columns in a single transaction.

Purpose: The Supabase JS client has no explicit transaction API — this Postgres function is the only way to guarantee that 4 rows (1 INSERT + 3 UPDATEs) are written atomically. If any step fails, the entire transaction rolls back. Without this function, the Edge Function cannot safely write donations.

Output: A PL/pgSQL function in the `treasury` schema, confirmed callable via the Supabase SQL editor with a test invocation (using a dry-run that rolls back).
</objective>

    Write the Postgres function SQL to `.planning/phases/03-webhook-backend/postgres-function.sql`.

    The function signature (from Phase 2 technical contract):
    - Schema: `treasury`
    - Name: `record_givebutter_donation`
    - Parameters:
      - `p_external_id TEXT` — GiveButter transaction ID (data.id from webhook payload)
      - `p_leaf_category_id UUID` — "Give Butter" category (depth=1), resolved by Edge Function
      - `p_parent_category_id UUID` — "Donations" category (depth=0), resolved by Edge Function
      - `p_budget_id UUID` — revenue budget ID for the EV fiscal year
      - `p_description TEXT` — constructed from donor name ("Donation from {first} {last}" or "GiveButter donation")
      - `p_amount NUMERIC` — donation amount (unit: confirm with $1 test; write function assuming dollars)
      - `p_date DATE` — from transacted_at, sliced to YYYY-MM-DD
      - `p_vendor TEXT` — hardcode 'GiveButter' from Edge Function
    - Returns: VOID
    - Language: plpgsql

    Function body must:
    1. IF EXISTS check: if a row already exists in treasury.budget_line_items WHERE external_id = p_external_id AND source = 'givebutter_webhook' → RETURN immediately (idempotency no-op, first guard)
    2. INSERT into treasury.budget_line_items with ALL these columns: category_id, description, approved_amount, actual_amount, vendor, date, external_id, source. Set source = 'givebutter_webhook'. Set both approved_amount and actual_amount to p_amount.
    3. UPDATE treasury.budget_categories SET amount = amount + p_amount WHERE id = p_leaf_category_id
    4. UPDATE treasury.budget_categories SET amount = amount + p_amount WHERE id = p_parent_category_id
    5. UPDATE treasury.budgets SET total_budget = total_budget + p_amount WHERE id = p_budget_id

    Use `CREATE OR REPLACE FUNCTION` so re-running the SQL is safe.

    Also write a DRY-RUN test block in the same file (separated by a comment) that wraps a test call in BEGIN/ROLLBACK so it can be verified without committing:

    ```sql
    FROM treasury.budget_line_items

    ROLLBACK;
    ```
