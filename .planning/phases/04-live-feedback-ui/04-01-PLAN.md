  <name>Task 1: Create useAnimatedCounter hook</name>
  <files>src/hooks/useAnimatedCounter.ts</files>
  <action>
    Create a new file `src/hooks/useAnimatedCounter.ts` that exports a single hook `useAnimatedCounter`.

    Signature:
    ```ts
    export function useAnimatedCounter(
      target: number,
      duration?: number,        // default 600 (ms) — matches CONTEXT decision
      onComplete?: () => void,
    ): number
    ```

    Behavior:
    - Tracks the current displayed value in `useState<number>(target)` (initialize to `target` so first render is not 0 → target on mount).
    - Uses `useRef<number>(target)` to remember the previous `target` across renders.
    - In a `useEffect` keyed on `[target, duration, onComplete]`:
      - If `target === prevRef.current`, do nothing (no animation, silent).
      - Otherwise, capture `startValue = prevRef.current`, `delta = target - startValue`, `startTime = performance.now()`.
      - Run a `requestAnimationFrame` loop that computes `t = Math.min(1, (now - startTime) / duration)`, applies `easeOutCubic(t) = 1 - Math.pow(1 - t, 3)`, and calls `setDisplay(startValue + delta * eased)`.
      - When `t >= 1`, set display to exact `target`, update `prevRef.current = target`, call `onComplete?.()`, and stop the loop.
    - Cleanup: on effect cleanup, cancel any in-flight `requestAnimationFrame` via `cancelAnimationFrame` and the stored frame id in a ref.

    CRITICAL pitfall (from research finding #6): The `onComplete` callback goes in the effect's dep array. If the consumer passes an inline arrow it will be recreated every render and the animation will reset. Document this at the top of the file with a JSDoc `@remarks` note: "Callers MUST wrap `onComplete` in `useCallback` with stable deps, otherwise the animation will restart on every render." Also verify by eye that `duration` has a default so callers typically pass only `target` + `onComplete`.

    Imports: `useEffect`, `useRef`, `useState` from 'react'. No other imports, no external libs.

    Do NOT:
    - Do not use `setInterval` — frame-synced animation only.
    - Do not use any animation library (no framer-motion, no gsap) — research confirmed no new packages.
    - Do not pin the hook to currency formatting; it returns a raw `number` and callers format it.
  </action>

    Rationale (from CONTEXT.md decisions): "Fire once per page load only — do not re-trigger on repeated tab switches."

    Step 3 — Inside the `App` function body, add a `handleDonateClick` callback using `useCallback`. Place it near the other `useCallback` handlers (search for `handleEntityChange` at line ~301 for a sibling location):

    ```tsx
    const handleDonateClick = useCallback(() => {
      if (donationRefetchArmed) return;
      donationRefetchArmed = true;

      const handleVisibility = () => {
        // CRITICAL pitfall from RESEARCH finding #2:
        // visibilitychange fires BOTH when hiding and when returning — guard for visible.
        if (document.visibilityState !== 'visible') return;
        // Remove ourselves FIRST so re-entry (alt-tab loops) can't double-fire.
        document.removeEventListener('visibilitychange', handleVisibility);

        // Guard: entity must still have a revenue dataset for the current year.
        // (Research finding #9: entity has available_datasets with fiscal_year + dataset_type.)
        if (!selectedEntity) return;
        const yearNum = parseInt(selectedYear);
        const hasRevenue = selectedEntity.available_datasets.some(
          d => d.fiscal_year === yearNum && d.dataset_type === 'revenue'
        );
        if (!hasRevenue) return;

        // Silent background refetch — no loading state, no spinner.
        clearCache();
        loadBudgetData(yearNum, selectedEntity.name, selectedEntity.state, 'revenue')
          .then(data => { setRevenueData(data); })
          .catch(err => { console.error('Post-donation revenue refetch failed:', err); });
      };

      document.addEventListener('visibilitychange', handleVisibility);
    }, [selectedEntity, selectedYear]);
    ```

    Pitfall addressed (research finding #8): `selectedEntity` and `selectedYear` MUST be in the `useCallback` deps or the closure captures stale values.

    CRITICAL pitfall addressed (research finding #2): Do NOT use `{ once: true }` on `addEventListener` — that would fire on the first hide event too. Manual removal inside the handler, gated on `visibilityState === 'visible'`, is the only correct pattern.

    Step 4 — Attach `handleDonateClick` to the Donate `<a>` at line ~576-587. Add `onClick={handleDonateClick}` to the anchor's props:

    ```tsx
    <a
      href="https://givebutter.com/g3e9u9"
      target="_blank"
      rel="noopener noreferrer"
      onClick={handleDonateClick}
      className="..."
      title="Donate to Empowered Vote"
    >
    ```

    Do NOT:
    - Do not add any spinner, loading flag, toast, or "checking for update" text. CONTEXT.md explicitly locks the no-change state to "completely silent." No visible side effect on refetch.
    - Do not preventDefault on the click — the `<a>` must still open GiveButter in a new tab.
    - Do not use `window` 'focus' event; CONTEXT.md locks this to `visibilitychange` because it works on mobile where `focus` is unreliable.
    - Do not put the armed-flag in component state (`useState`) — it must survive re-renders without causing them. A module-level `let` is the correct scope (persists for the page's lifetime, resets on hard reload).
    - Do not attach the listener on mount — it must be attached only AFTER the Donate click.
